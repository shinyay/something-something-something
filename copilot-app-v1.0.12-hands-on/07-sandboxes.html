クラウド / ローカル サンドボックスを体験するExperience cloud / local sandboxes

エージェントが本番環境に触れず、境界のあるローカルまたは GitHub ホストのクラウド環境でコードを実行・検証できる流れを体験します。Experience how agents can run and verify code in bounded local or GitHub-hosted cloud environments without touching production.

この機能でできることWhat you'll experience

体験のゴールYour goal

このガイドを終えると、ローカルとクラウドのサンドボックスを使い分け、制限ポリシーのもとでエージェントのコード検証を安全に進められるようになります。By the end of this guide you will be able to choose between local and cloud sandboxes and safely verify agent code under enforced policies.

境界のある実行場所Bounded execution places

本番環境ではなく、制限された場所でコードを動かす考え方を掴む。Understand the idea of running code in a restricted place instead of production.

ローカル制限の確認Local restriction checks

ファイルシステム・ネットワーク・システム機能へのアクセスが制限される様子を見る。See filesystem, network, and system access being restricted.

ephemeral なクラウド検証Ephemeral cloud verification

GitHub ホストの分離 Linux 環境で検証し、作業後に残さない運用を確認する。Verify in an isolated GitHub-hosted Linux environment that is discarded after use.

ハンズオン手順Step-by-step

サンドボックスで安全に検証するVerify safely in sandboxes

上から順に進めてください。各ステップは数分で完了します。Follow the steps in order — each takes just a few minutes.

  1. 準備Prepare

    検証用の小さな変更を用意するPrepare a small change to verify

    本番データや外部サービスに触れないリポジトリで、テストやビルドを実行できる小さな変更を用意します。In a repository that does not touch production data or external services, prepare a small change that can run tests or a build.

  2. ローカルLocal

    ローカル サンドボックスで実行するRun in a local sandbox

    エージェントの実行場所としてローカル サンドボックスを選び、制限されたファイルシステム・ネットワーク・システムアクセスの範囲でコード検証を開始します。Choose a local sandbox as the agent execution place and start code verification within restricted filesystem, network, and system access.

    • ガイド 02 のように複数エージェントを使う場合も、各作業が境界内で動くことを意識します。When using multiple agents as in guide 02, keep in mind that each task runs inside a boundary.
  3. ポリシーPolicy

    制限が強制されることを観察するObserve policy enforcement

    許可されていないファイルやネットワーク先に触れようとする操作は、ポリシーによりブロックまたは確認対象になります。どの制限が組織で管理されているかを記録します。Operations that try to touch disallowed files or network destinations are blocked or require review by policy. Note which restrictions are centrally managed by your organization.

  4. クラウドCloud

    GitHub ホストのクラウド サンドボックスへ切り替えるSwitch to a GitHub-hosted cloud sandbox

    同じ検証を、GitHub がホストする完全分離の ephemeral な Linux 環境で実行します。ローカルマシンに依存しない実行場所として使えることを確認します。Run the same verification in a fully isolated, ephemeral Linux environment hosted by GitHub. Confirm that it works as an execution place independent of your local machine.

    • クラウドセッションの引き継ぎはガイド 03 と組み合わせると理解しやすくなります。This pairs well with guide 03 when you want to understand cloud-session handoff.
  5. 確認Verify

    本番環境に触れていないことを確認するConfirm production was untouched

    テスト結果、変更ファイル、ログを確認し、本番データ・本番ネットワーク・永続的な外部状態に影響していないことを確認します。Review test results, changed files, and logs to confirm that production data, production networks, and persistent external state were not affected.

確認ポイントWhat you should see

うまくいったサインSigns it worked

チェックリストChecklist

  • ローカル サンドボックスで、許可されたリポジトリ範囲だけが読み書き対象になっている。In the local sandbox, only the allowed repository scope is readable or writable.
  • 制限外のファイル・ネットワーク・システム操作は、ブロックまたは確認対象として扱われた。Out-of-scope file, network, or system operations were blocked or surfaced for review.
  • クラウド サンドボックスは GitHub ホストの一時的な Linux 環境として実行され、検証後に残らない前提で使えた。The cloud sandbox ran as a GitHub-hosted ephemeral Linux environment that can be used without leaving state after verification.
  • ポリシーの定義元が個人設定ではなく組織側であることを確認できた。You confirmed that policies are defined by the organization rather than only by personal settings.
ヒント・応用Tips & variations

もう一歩使いこなすGo a little further

試してみるThings to try

  • 並列エージェント開発（ガイド 02）では、エージェントごとに検証範囲を小さくし、サンドボックスの境界を明確にします。For parallel agent development (guide 02), keep each agent's verification scope small and make sandbox boundaries explicit.
  • クラウドセッション（ガイド 03）と組み合わせると、別デバイスからでも同じ分離環境の結果を確認しやすくなります。Combine this with cloud sessions (guide 03) to make results from an isolated environment easier to review from another device.
  • 外部サービスが必要な検証は、実サービスではなくモックや読み取り専用のテスト用リソースを優先します。When verification needs external services, prefer mocks or read-only test resources over real production services.
サンドボックスは「何を許すか」を明確にする機能です。便利さよりも先に、組織のポリシーで守る境界を確認してから使い始めます。Sandboxes make explicit what is allowed. Before optimizing for convenience, confirm the boundaries protected by your organization's policies.
