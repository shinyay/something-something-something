Chats（旧 Quick Chats）を体験するExperience Chats (formerly Quick Chats)

リポジトリを選ばずにチャットセッションを開始し、/yolo や /allow-all-tools、セッションのフォーク、コードセッション相当の機能を試します。Start a chat session without choosing a repository, then try /yolo, /allow-all-tools, session forking, and code-session-level capabilities.

この機能でできることWhat you'll experience

体験のゴールYour goal

このガイドを終えると、リポジトリ未選択の状態から Chats を始め、必要に応じてツール承認やフォークを使い分けられるようになります。By the end of this guide you'll be able to start Chats without selecting a repository, then use tool approval and forks when they fit the conversation.

すぐ話し始めるStart immediately

リポジトリを決める前に、調査、設計相談、作業分解などの会話を開始します。Begin research, design discussion, or task breakdown before choosing a repository.

承認を調整するTune approvals

/yolo と /allow-all-tools を使い、ツール利用の承認を会話の目的に合わせて調整します。Use /yolo and /allow-all-tools to align tool approvals with the conversation goal.

会話を分岐するFork the conversation

セッションフォークで別案を試し、元の会話を壊さずに方向性を比較します。Fork a session to try an alternative direction without disrupting the original conversation.

始める前にBefore you begin

ハンズオン手順Step-by-step

リポジトリなしで Chats を始めるStart Chats without a repository

上から順に進めてください。各ステップは数分で完了します。Follow the steps in order — each takes just a few minutes.

  1. 開始Start

    リポジトリを選ばずにチャットを作るCreate a chat without choosing a repository

    Copilot App で新しいチャットセッションを開始します。最初にリポジトリを選ぶ必要はありません。調査したいこと、設計したいこと、整理したい作業をそのまま入力します。In the Copilot App, start a new chat session. You do not need to choose a repository first. Enter the research question, design topic, or work you want to organize.

  2. 相談Discuss

    通常のコードセッションと同じように依頼するAsk as you would in a code session

    要件整理、実装方針、コマンドの検討、ファイル操作を伴う作業の計画などを依頼します。必要になったら、後からリポジトリやファイルの文脈を追加します。Ask for requirements clarification, implementation planning, command planning, or a plan for work that may involve files. Add repository or file context later when you need it.

  3. 承認Approve

    /yolo で承認の手間を減らすUse /yolo to reduce approval friction

    信頼できる範囲の作業で、ツール利用の確認を減らしたい場合は /yolo を入力します。影響範囲が大きい作業では、実行前に内容を確認できる運用に戻します。For trusted work where you want fewer tool-use confirmations, enter /yolo. For higher-impact work, switch back to a workflow where you review actions before they run.

  4. 許可Allow

    /allow-all-tools を必要な場面だけ使うUse /allow-all-tools only when needed

    複数のツールをまとめて使う調査や検証では /allow-all-tools が役立つ場合があります。安全性が気になる会話では、許可する範囲を狭く保ちます。For research or validation that needs several tools, /allow-all-tools can help. In conversations where safety matters, keep the allowed scope narrower.

  5. 分岐Fork

    セッションをフォークして別案を試すFork the session to try another path

    会話の途中で別の方針を試したくなったら、セッションをフォークします。元の流れを残したまま、新しいチャットで別案、別の制約、別の実装方針を比較できます。When you want to try a different direction, fork the session. The original flow stays intact while a new chat explores another option, constraint, or implementation strategy.

チェックリストChecklist

  • リポジトリを選ばずにチャットセッションを開始できた。You started a chat session without selecting a repository.
  • /yolo または /allow-all-tools を入力し、ツール承認の挙動を調整できた。You entered /yolo or /allow-all-tools and adjusted tool-approval behavior.
  • 会話の途中からセッションをフォークし、別案を独立して試せた。You forked the session mid-conversation and tried an alternative independently.
  • 必要に応じてコードセッションと同等の相談・計画・実行支援ができることを確認できた。You confirmed that Chats can support discussion, planning, and execution assistance comparable to a code session when needed.
ヒント・応用Tips & variations

もう一歩使いこなすGo a little further

試してみるThings to try

  • リポジトリがまだ決まっていない調査、提案文の下書き、実装方針の比較は Chats から始めると速く進みます。For research before a repository is chosen, proposal drafts, or implementation tradeoffs, starting in Chats is often faster.
  • 長期的に残したい知見は Memory++ と組み合わせると便利です。詳しくはガイド 12 を確認してください。For knowledge you want to keep over time, combine Chats with Memory++; see guide 12.
  • 最終的に具体的なコード変更へ進む場合は、会話で整理した要件をリポジトリ付きのセッションへ渡すと手戻りを減らせます。When you move into concrete code changes, pass the clarified requirements into a repository-backed session to reduce rework.
Chats は「リポジトリを選ぶ前の入口」です。素早く始め、必要になった時点で承認範囲、フォーク、コード文脈を足していくと使いやすくなります。Chats are the entry point before you choose a repository. Start quickly, then add approval scope, forks, and code context only when you need them.
